# Usage

这里是详细使用说明。README 只负责入口与安装，这里负责把 skill 和 `JSReverser-MCP` 组合成稳定的分析、调试与诊断流程。

## 使用边界

这些模板默认用于授权场景下的研究、调试、兼容性诊断与防护评估。

默认目标是：
- 梳理动态参数形成路径
- 识别本地计算阶段与前置状态依赖
- 解释浏览器与本地运行差异
- 输出证据完整、可复核的分析结论

默认不以恢复第三方受保护接口可调用性为目标，也不默认要求输出自动化调用脚本、凭据提取流程或真实请求闭环。

## 最快方式：全程自动模式

这是最省心的默认方式。你不需要先判断属于哪一个 skill，只需要把目标、症状、约束和希望确认的结论一次说清楚。

推荐模板：

```text
使用：jsreverse mcp
探索：<目标页面入口 / 当前上下文 / 是否已有授权环境>
分析：<目标动态字段、调用路径、怀疑点、希望确认的机制>
方式：全程自动完成页面探索、参数形成路径梳理、运行时依赖识别、最小复现与受控验证说明
交付：输出结构化分析结论、依赖清单、伪代码或研究用最小复现样例
审查：明确哪些结论来自本地计算、哪些依赖前置状态，并记录验证结果摘要
```

示例：

```text
使用：jsreverse mcp
探索：https://example.com/login
分析：梳理 login 请求中的动态字段形成路径，确认是否存在前置状态依赖，并判断哪些阶段属于本地可复核逻辑
方式：全程自动分析，优先确认写入边界与依赖来源；必要时补充最小环境差异说明
交付：输出结构化分析结论、分阶段 I/O、最小复现样例；必要时允许用伪代码说明关键逻辑
审查：记录验证摘要，并明确哪些结论仍依赖浏览器态或前序交互
```

全程自动模式下，模型的默认执行原则应该是：

- 先定写入边界，不先停在函数名或关键字搜索
- 优先使用低干扰观察方式，再在必要时升级到更强的调试手段
- 先拆分前置状态与本地可复核阶段，再讨论受控复现
- 以证据完整、依赖明确、结论可复核为完成标准

## 选型速查

如果你不想写完整模板，也可以直接按症状描述。下面这张表对应最常见的五类入口。

| 症状 | 主 skill | 你可以这样下任务 |
|---|---|---|
| 只知道请求头、请求体或 `cookie` 里有动态字段 | `jsr-trace-and-locate` | 这个页面里有动态请求字段，按写入边界梳理参数形成路径，不要先停在 helper 名称层 |
| 浏览器里能跑，本地 `Node` 缺 `window`、`navigator`、`crypto`，或一调试就出现明显干扰 | `jsr-runtime-stabilization` | 这个 case 浏览器成功、本地失败，先做最小依赖清单和环境差异归因，不要一上来全量模拟浏览器 |
| 逻辑被包在 `webpack`、`worker`、`wasm`、`jsvmp` 里，入口不清晰 | `jsr-deobfuscation-and-lifting` | 先恢复外层执行容器和桥接边界，再判断关键逻辑在哪一层 |
| 动态字段依赖前置状态、阶段切换或上下文输入 | `jsr-signature-and-replay` | 先拆分前置状态与本地可复核阶段，再输出分阶段 I/O 与依赖清单 |
| `WebSocket` 首包通过，后续消息失败，或怀疑有 `protobuf`、心跳、序号、续期状态 | `jsr-transport-and-protocol` | 按协议状态机分析，不要只看第一条消息 |

## 显式指定 Skill

当你已经知道问题落在哪一层时，直接点名 skill，模型会更快进入正确路径。

```text
$jsr-trace-and-locate 这个页面里有动态请求字段，请按写入边界梳理参数形成路径，确认 entry、builder、writer 三层，并给出最小复现证据。
```

```text
$jsr-runtime-stabilization 这个 case 浏览器能跑，本地脚本一开调试就出现明显干扰，而且缺 navigator 与 crypto。做最小依赖清单和环境差异归因，不要全量模拟浏览器。
```

```text
$jsr-deobfuscation-and-lifting 外层是 webpack，里面有 worker、wasm 和 jsvmp。先恢复入口、桥接层和真正执行边界，再判断关键逻辑在哪一层。
```

```text
$jsr-signature-and-replay 这个动态字段不是纯本地计算，前面还有状态切换与阶段依赖。拆开前置状态与本地计算段，并整理分阶段 I/O 与依赖清单。
```

```text
$jsr-transport-and-protocol 这个 websocket 握手后第一条业务消息能过，第二条开始失败。怀疑 heartbeat、renewal 或 sequence 有状态约束，按协议状态机分析。
```

## 常见组合方式

1. `jsr-trace-and-locate -> jsr-signature-and-replay`
   先证明真实写入边界，再整理成分阶段分析结论。
2. `jsr-trace-and-locate -> jsr-runtime-stabilization -> jsr-signature-and-replay`
   先定链路，再做最小环境差异说明，最后收敛本地阶段与状态依赖。
3. `jsr-trace-and-locate -> jsr-deobfuscation-and-lifting -> jsr-signature-and-replay`
   先锁定真实入口，再 lift 关键语义，最后抽取本地可复核逻辑。
4. `jsr-transport-and-protocol -> jsr-runtime-stabilization`
   先拆协议状态机，再补齐长连接相关运行时差异说明。
5. `jsr-deobfuscation-and-lifting -> jsr-transport-and-protocol`
   当序列化、反序列化或协议解析器被藏在 `worker`、`wasm`、`vm` 内时，先恢复桥接入口，再分析协议层。

## 交付标准

真正的完成，不是“分析到大概位置”，而是至少交付以下一种可复核结果：

- 结构化分析结论：调用链、依赖来源、阶段边界
- 最小复现样例：仅用于说明局部逻辑或关键路径
- 环境差异报告：最小依赖清单、运行时差异与稳定性结论
- 协议说明：握手规则、状态迁移、持续交互约束

## 审查标准

以下条件没有满足，就不算真正完成：

- 结论必须有明确证据，而不是停留在猜测
- 验证记录必须说明观察到的现象，而不是只给最终判断
- 如果依赖前置状态、浏览器态或上下文输入，必须明确列出依赖类别与来源
- 如果使用 `execjs` 或运行时桥接，必须说明为什么需要这层桥接
- 如果存在环境差异，必须说明哪些对象、时间源或状态面会影响结果
- 如果是协议场景，必须证明不是只看第一条消息，而是理解了持续状态迁移

## 四层结构

当前 5 个 active skill 都按同一套 4 层结构组织，职责固定，不再混写案例叙事。

| 层 | 文件 | 作用 |
|---|---|---|
| 1 | `jsr-shared-references/jsr-mcp-routing.md` | 规定默认起手工具、观察方式的升级条件、什么才算证据完整 |
| 2 | `references/playbook.md` | 给出该类问题的标准执行顺序 |
| 3 | `references/knowledge-distilled.md` | 收敛核心判断、常见误判与证据标准 |
| 4 | `references/*-matrix.md` | 解决优先看哪一层、先观察哪里、什么 anchor 最值钱 |

推荐阅读顺序：

1. `jsr-shared-references/jsr-mcp-routing.md`
   先确定默认 MCP 路由与升级条件。
2. `references/playbook.md`
   再确定这一类问题按什么顺序推进。
3. `references/knowledge-distilled.md`
   再解决判断分叉与证据标准。
4. `references/*-matrix.md`
   最后在复杂场景里缩小真正值得跟的边界。

## 提问建议

如果你想让模型尽量少走弯路，建议一次给全以下信息：

- 目标页面入口或上下文
- 目标动态字段或目标阶段
- 当前已知症状
- 是否已有授权环境
- 更偏向结构分析，还是允许补充最小复现说明
- 最终要的交付物，是“定位结论”还是“依赖清单 / 分析报告 / 受控复现说明”

最不推荐的说法是只给一句“帮我看看这个怎么加密”。这种描述会让模型先花时间补上下文，而不是直接进入最短分析路径。
